ФАЗА 1 — P0: Критичні баги, які ламають функціонал (робити першими, без винятків)

1.1. Баг: невірний job ID у worker.on("failed") → джоба ніколи не отримує статус failed

Файл: apps/worker/src/index.ts

Проблема: У обробнику worker.on("failed", async (job, err) => {...}) використовується job.id (внутрішній інкрементальний ID BullMQ) замість job.data.jobId (реальний Prisma UUID). В результаті prisma.job.update({ where: { id: job.id } }) кидає P2025 RecordNotFound, ловиться мовчки в catch, і джоба навіки залишається в проміжному статусі (наприклад analyzing), а мобільний клієнт нескінченно полить (refetchInterval ніколи не зупиняється).

Що зробити:


Знайти весь блок worker.on("failed", async (job, err) => {...}).
Замінити всі звернення job.id на job.data.jobId там, де мова йде про Prisma-запис (в prisma.job.update і prisma.jobEvent.create).
Залиш job.id (BullMQ id) тільки в логах, де це семантично коректно (наприклад, для дебагу самої черги), але додай окремо також job.data.jobId в лог-об'єкт, щоб не було плутанини — перейменуй поле логу з jobId: job.id на bullJobId: job.id, dbJobId: job.data.jobId.
Зроби те саме звірку в worker.on("completed", ...) — там зараз коментар "Already logged inside job", але перевір, чи там теж немає прихованого використання job.id замість job.data.jobId в майбутньому коді, який хтось додасть. Онов коментар: // Status already updated inside processor via job.data.jobId — do not duplicate DB writes here.
Захисний механізм (defense in depth): додай на початку функції-процесора (async (job: BullJob) => {...}) явну перевірку:


ts   if (!job.data.jobId) {
     throw new Error(`Job payload missing jobId (bullJobId=${job.id})`);
   }

Це не дасть у майбутньому тихо зіпсувати дані, якщо mediaQueue.add колись викличуть без jobId в payload.

Acceptance Criteria:


Написати мінімальний ручний/скриптовий тест: примусово зіпсувати originalUrl (наприклад, підставити неіснуючий S3-ключ) так, щоб усі 3 attempts BullMQ провалились → перевірити в БД, що Job.status === 'failed' і Job.error заповнене, а не залишилось у проміжному статусі.
grep -rn "where: { id: job.id }" apps/worker/src повертає 0 результатів.



1.2. Баг/вразливість: відсутність контролю розміру файлу в Presigned URL

Файли: apps/web-api/services/s3.service.ts, apps/web-api/app/api/upload-url/route.ts

Проблема: PutObjectCommand в generateUploadUrl не обмежує розмір файлу. Presigned PUT сам по собі не має вбудованого ліміту.

Що зробити:


Додати константи максимального розміру за типом файлу у спільний конфіг, наприклад apps/web-api/lib/upload-limits.ts:


ts   export const MAX_IMAGE_BYTES = 25 * 1024 * 1024;  // 25 MB
   export const MAX_VIDEO_BYTES = 250 * 1024 * 1024; // 250 MB


У UploadSchema (app/api/upload-url/route.ts) додати поле fileSize: z.number().int().positive() — клієнт зобов'язаний передавати очікуваний розмір файлу.
У роуті — валідувати fileSize проти ліміту залежно від contentType.startsWith("image/") чи video/". Якщо перевищено — повертати 413.
У S3Service.generateUploadUrl передавати ContentLength в PutObjectCommand АБО (кращий варіант, бо ContentLength в presigned PUT команду не завжди enforce-иться S3 надійно) — перейти на createPresignedPost (S3 POST Policy) з умовою content-length-range:


ts   import { createPresignedPost } from "@aws-sdk/s3-presigned-post";

   const { url, fields } = await createPresignedPost(s3Client, {
     Bucket: s3BucketName,
     Key: key,
     Conditions: [
       ["content-length-range", 0, maxBytes],
       ["eq", "$Content-Type", contentType],
     ],
     Fields: { "Content-Type": contentType },
     Expires: 3600,
   });


Важливо: якщо переходиш на createPresignedPost, зміниться контракт API-відповіді ({ url, fields } замість { uploadUrl }) — синхронно онови мобільний клієнт apps/mobile/features/upload/hooks/useMediaUpload.ts, де зараз FileSystem.createUploadTask(urlData.uploadUrl, imageUri, { httpMethod: 'PUT', ... }). POST-policy вимагає multipart/form-data запиту, а не PUT — це НЕ тривіальна заміна одного рядка, плануй окремий рефакторинг upload-хука під multipart form upload (Expo FileSystem.uploadAsync з httpMethod: 'POST', parameters: fields, fieldName: 'file').
Якщо часу на повний перехід на POST Policy немає — мінімальний фікс: залишити PUT, але додати ContentLength в PutObjectCommand, і обов'язково додати серверну перевірку в S3 через HeadObjectCommand одразу після завантаження (перед постановкою в чергу, в POST /api/jobs) — перевірити ContentLength реального об'єкта в S3 і відхилити job (+ видалити файл), якщо перевищено ліміт.


Acceptance Criteria:


Спроба залити файл понад ліміт повертає явну помилку (413 на етапі запиту upload-url, АБО job відхиляється з поясненням при перевищенні реального розміру в S3).
Мобільний upload-флоу як і раніше показує прогрес і успішно завершується для файлів у межах ліміту.

2.2. Race condition в rate-limiter (POST /api/jobs)

Файл: apps/web-api/app/api/jobs/route.ts

Що зробити:


Замінити:


ts   const currentCount = await connection.incr(rateLimitKey);
   if (currentCount === 1) await connection.expire(rateLimitKey, 60);

на атомарну Lua-операцію через ioredis (connection.defineCommand або inline eval):

ts   // lib/rate-limit.ts
   const RATE_LIMIT_SCRIPT = `
     local current = redis.call("INCR", KEYS[1])
     if tonumber(current) == 1 then
       redis.call("EXPIRE", KEYS[1], ARGV[1])
     end
     return current
   `;
   export async function checkRateLimit(connection: Redis, key: string, windowSeconds: number, limit: number) {
     const current = await connection.eval(RATE_LIMIT_SCRIPT, 1, key, windowSeconds) as number;
     return { allowed: current <= limit, current };
   }


Використати цю функцію в POST /api/jobs замість інлайн-логіки.
Винести rate-limit в перевикористовуваний middleware withRateLimit(handler, { keyPrefix: "jobs", limit: 5, windowSeconds: 60 }), який можна буде застосувати і до інших ендпоінтів (див. 2.3).


Acceptance Criteria:


Юніт-тест (Фаза 5) на checkRateLimit, що емулює конкурентні виклики й перевіряє коректний TTL.
grep підтверджує відсутність старого INCR/if (currentCount === 1) патерну в коді.

      // Використати router.replace через імпорт expo-router на верхньому рівні модуля непросто (немає контексту навігації тут) —
      // тому: емітувати кастомну подію/колбек, на яку підпишеться root layout (_layout.tsx), і він викличе router.replace("/(auth)/login")
      authEventEmitter.emit("session-expired");
    }
    throw new Error(error.message || "API Error");
  }
  return data;
};

4.4. Атомарність видалення job (БД + S3)

Файл: apps/web-api/services/job.service.ts

Проблема: видалення запису в БД і видалення файлів S3 виконуються не атомарно — можливі сирітські файли.

Що зробити (прагматичний компроміс, без розподілених транзакцій):


Змінити порядок операцій: спершу спробувати видалити файли з S3, і лише при успіху — видаляти запис БД. Якщо видалення S3 частково не вдалось (наприклад, processedUrl видалився, а originalUrl — ні) — не переривати процес критично, але:
Додати "м'яке видалення" (soft delete) як проміжний крок: позначити Job полем deletedAt (нове поле в схемі) одразу, потім асинхронно (або синхронно в тому ж запиті) видаляти S3-файли, і тільки після підтвердження видаляти запис остаточно (hard delete) або залишити soft-deleted записи, відфільтровані з getJobs/getJobDetails (where: { deletedAt: null }).
Мінімальна версія без soft-delete: обгорнути видалення S3-файлів у Promise.allSettled, залогувати будь-які невдачі з чітким event: "ORPHANED_S3_FILE" (щоб було що моніторити), і тільки після цього видаляти запис БД. Це не гарантує 100% атомарність, але прибирає легко-виправну проблему "спочатку видаляємо БД, потім S3" (зараз навпаки — S3 видаляється першим, що взагалі-то безпечніше, ніж описано в рев'ю; перевір актуальний порядок операцій у коді перед фіксом і збережи S3-first підхід, лише додай явне логування помилок видалення S3 через Promise.allSettled замість мовчазного try/catch в deleteFile).


Acceptance Criteria: якщо видалення одного з двох S3-об'єктів штучно провалюється (замокати), інформація про це логується з чітким тегом, і запис БД видаляється лише після спроби прибрати обидва файли.
